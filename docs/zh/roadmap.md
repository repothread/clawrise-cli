# Clawrise OSS Roadmap

## 1. 这份文档想说明什么

这份 roadmap 面向用户、贡献者和关注项目方向的社区读者。

它关注的是 Clawrise 在接下来一段时间里最值得优先做好的事情，而不是把已经完成的功能重新罗列一遍。  
已经落地的能力，请以项目 `README` 中的 `当前状态` 为准。

## 2. Clawrise 当前在做什么

Clawrise 仍然会坚持自己的核心定位：

- 它是一个面向 agent 和自动化场景的多平台 CLI 执行层。
- 它关注的是稳定的任务执行、统一的调用方式，以及可持续扩展的平台接入模型。
- 它不是某一个 SaaS 平台的专属 CLI，也不准备把自己做成“所有平台 CLI 的大一统替代品”。

对当前开源版本来说，最重要的判断是：

- 我们会优先为“API 完整、自动化价值高、但 CLI 生态薄弱或空白”的 SaaS 平台提供更好的命令行能力。
- 我们会继续保持 plugin-first 的方向，让平台能力通过 provider plugin 扩展，而不是重新硬编码回 core。
- 我们会对已经存在成熟 CLI 的平台保持开放、积极、尊重生态的态度，但不会把“重写并对撞完整命令面”作为近期主线。

这意味着：

- 像 Notion 这类“有 API、但 CLI 空白明显”的平台，仍然非常适合成为开源版本的重点方向。
- 像 Feishu 这类已经存在成熟 CLI 的平台，Clawrise 会继续关注，也会保留必要的第一方能力与验证样本，但不会把近期目标放在覆盖率竞争上。

## 3. 接下来一段时间，我们最想先做好什么

### 3.1 让安装、升级和校验路径更清晰

Clawrise 已经进入 plugin-first 阶段，但第一方 plugin 的正式交付路径还需要继续打磨。

接下来我们会优先补齐：

- 更清晰的第一方 plugin 发布产物与版本结构
- 更容易理解的安装、升级与兼容性检查路径
- 对 core 与 plugin 兼容关系更直观的说明
- 更可靠的 `plugin verify` 使用体验

截至当前仓库，这一方向已经完成了第一轮关键收口：

- 已支持 `plugin upgrade` 与 `plugin upgrade --all`
- 已把 `protocol_version` / `min_core_version` 的兼容性检查前移到安装与升级阶段
- 已让 `plugin verify` 输出结构化 trust 结果
- 已支持通过 `registry_source` resolve 结果继续完成 `registry://` 安装

后续这条线更偏向 hardening，而不是再从零补 capability 骨架。

我们希望用户看到的是：

- 第一次安装第一方 plugin 时，路径更短、更明确
- 升级时能更早发现兼容性问题，而不是等到真实执行时才踩坑

### 3.2 让第一次跑通真实调用更容易

目前命令面已经具备一定能力，但从 fresh install 到第一次真实成功调用，路径仍然偏长。

这一阶段我们会继续压缩首次接入成本，重点包括：

- 更紧凑的 quickstart
- 更顺手的 `config init`、`auth check`、`doctor` 到真实执行路径
- 与真实 CLI 输入形状更一致的样例输入
- playbook 与可执行 operation 之间更直观的关联

我们的目标不是让用户先读完设计文档再上手，而是尽快让用户跑通第一条真实调用。

### 3.3 优先把精力放在 CLI 空白的平台上

Clawrise 开源版本的独立价值，首先来自“补足空白”。

因此，近期第一方 provider 的优先级会更偏向这些平台：

- API 已经足够可用
- 自动化场景明确
- 用户量或使用需求并不小
- 但 CLI 能力仍然薄弱，或者社区还没有形成成熟的事实标准工具

这类平台更适合 Clawrise 去投入，因为这里的新增价值足够清楚：

- 用户不只是多了一个包装层，而是真的多了一套可用命令面
- 社区贡献者也更容易理解项目为什么值得做
- 项目可以围绕这些平台沉淀稳定的 task-level operations、playbooks 和 metadata

### 3.4 对成熟 CLI，我们会保持开放和协作

我们也不想把项目做成一个封闭、对抗生态的工具。

如果某个平台已经存在成熟 CLI，Clawrise 的态度会是：

- 开放
- 积极
- 尊重已有生态
- 优先寻找互补关系，而不是默认进入正面对撞

这并不意味着忽视这些平台，而是意味着我们会更克制地处理它们：

- 不急着复制完整命令树
- 不把覆盖率竞赛当成近期目标
- 不让项目资源被单个平台的已有生态牵着走

更现实的方向是：

- 保留少量高价值、适合 agent 和跨任务场景的第一方原生能力
- 观察未来与成熟 CLI 协作、互补或兼容的机会
- 在公开文档中把这层态度写清楚，减少误解

### 3.5 让插件接入和扩展更容易理解

如果插件作者必须反向阅读大量 core 实现，生态增长会很慢。

因此，接下来我们会继续改善：

- plugin author guide
- manifest 与兼容性说明
- 本地校验与最小可用接入路径
- 更清晰的 handshake、catalog 和 execute 测试指引

我们的目标是让贡献者可以更快回答两个问题：

- 我能不能为某个平台写一个最小 provider plugin？
- 我写出来之后，怎么知道它和当前 core 是否兼容？

### 3.6 让 `spec`、completion 和文档继续共享同一份事实源

Clawrise 已经有了 `spec export` 和 completion，这条路线应该继续收紧，而不是分叉。

我们会继续推动：

- 更稳定的元数据导出契约
- 基于同一元数据层生成 operation reference
- 让 runtime facts、catalog、completion 和生成文档之间的关系更直观

这样做的价值很直接：

- 用户看到的能力说明更一致
- 贡献者不需要维护多套容易漂移的文档事实源
- agent 也更容易消费同一套稳定的发现面

## 4. 我们希望社区接下来能看到什么变化

如果这阶段的工作推进顺利，社区应该能明显感受到这些变化：

- 第一方 plugin 的安装、升级、校验路径比现在更清楚
- 新用户更容易从零跑通一次真实调用
- 项目的平台选择逻辑比现在更明确，更容易理解为什么某些平台优先、某些平台暂缓
- Clawrise 对成熟 CLI 的态度会更公开、更清晰，不会给人“要把所有成熟工具重做一遍”的印象
- 第三方插件作者会拥有更可依赖的接入文档和兼容性边界
- `spec`、completion 和 operation 文档之间的关系会更加统一

## 5. 后续值得继续推进的方向

下面这些方向依然有价值，只是优先级会放在前面那些重点之后。

### 5.1 继续扩展高信号 playbooks

我们会继续围绕现有 provider 补充更贴近真实任务的 playbook，尤其是：

- 高频任务
- 可验证路径清楚的任务
- 能体现 Clawrise task-level value 的任务

### 5.2 继续观察成熟 CLI 的协作机会

Feishu 不会是唯一存在成熟 CLI 的平台。

后续像 `gh` 这样的工具，也值得认真观察和研究。  
但这类方向更适合作为“互补与协作”的延伸，而不是优先级高于 CLI 空白平台的主战场。

### 5.3 扩展更多值得做的第一方 provider

未来继续扩 provider 是自然方向，但前提依然应该是：

- 值得做
- 用户确实需要
- 能形成稳定任务能力
- 不只是为了追求平台数量

## 6. 当前不优先的方向

这些方向并不是没有价值，只是现在不会排在最前面：

- 公共 plugin marketplace
- 不可信 plugin 的沙箱隔离
- REPL-first 交互壳
- 完整 JSON Schema 框架
- 跨平台 workflow engine
- 为了覆盖率而做的大而全平台专属命令树重写

## 7. 我们会尽量避免什么

为了保持项目方向稳定，接下来我们会尽量避免：

- 把平台细节重新硬编码回 core
- 把项目重新拉回“单平台 CLI 重写工程”
- 因为外部已有成熟生态，就急着投入无效的覆盖率竞争
- 重新制造多套元数据事实源，让 `spec`、docs、completion 和 runtime 继续漂移
- 在安装源已经支持远程的情况下，误以为 trust model 已经足够完整
- 在还没判断清楚某个平台是否值得做第一方 provider 之前，就过早铺开平台面

## 8. 小结

Clawrise 接下来的开源重点，不是把所有平台都抢先做一遍，而是把真正值得做的部分先做好。

近期更重要的是：

- 把 plugin-first 这条路打磨得更稳
- 把首次接入体验做得更短
- 把 CLI 空白平台补得更实
- 把对成熟 CLI 的态度讲得更清楚
- 把面向社区的扩展路径整理得更容易参与

如果这些部分逐步成型，Clawrise 的价值会更容易被用户和社区看见。

       }

    2. 修改 `newPluginSecretStore` 函数签名，接受 ctx 参数：
       ```go
       func newPluginSecretStore(ctx context.Context, manifest pluginruntime.Manifest) Store {
           return &pluginSecretStore{
               client: pluginruntime.NewProcessSecretStore(manifest),
               ctx:    ctx,
           }
       }
       ```

    3. 修改 `openNamedStore` 中所有调用 `newPluginSecretStore(manifest)` 的地方为 `newPluginSecretStore(context.Background(), manifest)`。有两处调用（line ~159 和 line ~169）。注意：使用 `context.Background()` 而非其他 ctx，因为 openNamedStore 本身不在 Store 接口的调用链中，且 Store 接口无法传递 context。这是 CTX-03 的最佳折中方案——保持了 Store 接口不变。

    4. 修改 `pluginSecretStore.Backend()` 方法，将 `context.Background()` 替换为 `s.ctx`：
       ```go
       descriptor, err := s.client.DescribeStorageBackend(s.ctx)
       ```

    5. 修改 `pluginSecretStore.Status()` 方法，将 `context.Background()` 替换为 `s.ctx`：
       ```go
       status, err := s.client.Status(s.ctx)
       ```

    6. 修改 `pluginSecretStore.Get()` 方法，将 `context.Background()` 替换为 `s.ctx`：
       ```go
       result, err := s.client.Get(s.ctx, pluginruntime.SecretStoreGetParams{
       ```

    7. 修改 `pluginSecretStore.Set()` 方法，将 `context.Background()` 替换为 `s.ctx`：
       ```go
       return s.client.Set(s.ctx, pluginruntime.SecretStoreSetParams{
       ```

    8. 修改 `pluginSecretStore.Delete()` 方法，将 `context.Background()` 替换为 `s.ctx`：
       ```go
       return s.client.Delete(s.ctx, pluginruntime.SecretStoreDeleteParams{
       ```

    关键设计决策（per RESEARCH.md CTX-03 分析）：保持 `Store` 接口不变，在 `pluginSecretStore` 内部存储 context 引用。构造时传入 `context.Background()` 作为默认 context。虽然 `context.Background()` 不可取消，但这解决了将来的扩展路径——当调用链中有 ctx 可用时，可在构造 pluginSecretStore 时传入可取消的 ctx，无需修改接口。
